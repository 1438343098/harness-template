# **产品需求文档（PRD）**

## **概述**

设计全面的、可直接投入生产的产品需求文档（PRD），填补业务愿景与技术执行之间的鸿沟。本技能适用于现代软件系统，确保需求被清晰定义。

## **何时使用**

在以下情况使用本技能：

- 启动新产品或功能开发周期
- 将模糊想法转化为具体技术规格
- 为 AI 功能定义需求
- 利益相关方需要统一的项目范围"唯一来源"
- 用户要求"写 PRD"、"整理需求"或"规划功能"

## **工作流程**

1. 接收用户的功能描述。
2. 根据文档需求，尽可能多地提问，确保所有必要字段都已填写完整。
3. 根据收集到的答案，按照下方严格 PRD 结构生成文档。
4. 保存到 **`docs/prd/[功能名称].md`**

## **严格 PRD 结构**

输出**必须**遵循以下精确结构：

### **1. 执行摘要**

- **问题陈述**：1–2 句话说明痛点。
- **解决方案与目标**：1–2 句话说明解法。
- **成功标准**：3–5 个可量化的 KPI。

### **2. 用户故事**

每个故事需包含：

- 标题：简短的描述性名称
- 描述："作为 [用户]，我希望 [功能]，以便 [收益]"
- 优先级：P0 / P1 / P2
- FR：关联的功能需求编号
- 验收标准：可验证的完成条件清单
- 每个故事应小到可在一次专注的开发会话中完成。

**格式：**

```
### US-001: [标题]
**优先级:** PN

**描述:** 作为 [用户]，我希望 [功能]，以便 [收益]。

**FR:** FR-001，FR-002

**验收标准:**
- [ ] 具体可验证的条件
- [ ] 另一个条件
```

**重要：**

验收标准必须可验证，不能模糊。"正常工作"是坏的。"点击删除按钮后显示二次确认弹窗"是好的。

### **3. 功能需求**

维护一个按编号排列的能力目录（`FR-001`、`FR-002`……）。每条记录必须**明确、可测试、无歧义**——不允许含糊带过。

**每条 FR 必须包含的子章节**

| 子章节 | 用途 |
| ------ | ---- |
| **描述** | 系统需要交付什么，一段简洁的说明（如有必要，说明范围的边界）。 |
| **关联用户故事** | 交叉引用故事编号（如 `US-001`、`US-002`）。 |
| **参与者** | 发起或受影响的角色：终端用户、管理员、子账号、主账号、系统/集成等。 |
| **前置条件** | 流程开始前必须满足的状态、权限或数据。 |
| **触发条件** | 启动流程的事件：用户操作（注明 UI 中的**具体位置**）、API 请求、定时任务、webhook、后台任务等。 |
| **主流程** | 按顺序编号的**正常路径**步骤。 |
| **分支流程** | 带标签的分支（`A1`、`A2`……）：错误、校验失败、取消、超时、权限拒绝、部分成功。 |
| **后置条件** | **成功**后的系统/数据状态；如失败结果不同，需单独说明。 |
| **验收标准** | 可验证的条件——优先使用每个场景的 **Given / When / Then** 格式，或 QA 可执行的等效清单。 |

**范围规则**

- **一个 FR = 一项内聚的能力。** 如果内容过大，拆分为 `FR-002`、`FR-003`……并明确边界。
- **UI 和触发条件必须可定位：** 命名屏幕/区域/控件及其产生的行为。*坏的：*"当用户点击按钮时，弹窗打开。" *好的：*"当用户点击列表行右下角的**删除**按钮时，打开一个带有**取消**和**删除**操作的**二次确认**弹窗。"
- **分支流程是必须的**，只要失败或变体在现实中是合理的（鉴权、网络、空状态、限制）。

**格式模板**

```markdown
## FR-001

### 描述

...

### 关联用户故事

US-001, US-002

### 参与者

...

### 前置条件

...

### 触发条件

...

### 主流程

1. ...
2. ...

### 分支流程

- **A1:** ...
- **A2:** ...

### 后置条件

...

### 验收标准

- **Given（前提）** ...
- **When（当）** ...
- **Then（则）** ...
```

**重要：**

- 文档必须明确需求的业务领域，单个需求只能作用于一个业务领域。例如：明确修改用户 profile 领域的需求，或明确修改云盘业务领域的 xxx 功能。
- 文档必须明确现状。
- 用户交互描述必须足够明确，请补充尽可能多的定语，以便 AI 和人类都能清晰理解。"当用户点击按钮时，弹窗"是坏的；"当用户点击列表组件右下角的"删除"按钮时，弹出二次确认弹窗"是好的。
- 文档必须明确需求变动范围。例如：修改 a 页面点击 b 按钮后展示出的 c 面板下方的提交按钮，将提交按钮移动到面板的顶部左侧。
- 文档必须明确需求的操作逻辑边界。例如：明确用户收藏的作品后，展示的逻辑是针对用户维度、企业维度还是全部人。

### **4. 设计考量（可选）**

- UI/UX 要求
- 如有原型图，附上链接
- 可复用的现有组件

### **5. 风险与路线图**

- **分阶段交付**：MVP → v1.1 → v2.0。
- **技术风险**：延迟、成本或依赖失效。

## **执行指南**

### **需求质量**

使用具体、可量化的标准。避免"快"、"简单"或"直观"等模糊词汇。

```
# 模糊（坏）
- 搜索应该快速并返回相关结果。
- UI 必须看起来现代且易于使用。

# 具体（好）
+ 搜索必须在 10k 条记录的数据集上 200ms 内返回结果。
+ 搜索算法在基准评测中必须达到 >= 85% 的 Precision@10。
+ UI 必须遵循 'Vercel/Next.js' 设计系统，并达到 100% 的 Lighthouse 无障碍分数。
```

### **必须做（始终）**

- **定义测试方式**：对于 AI 系统，说明如何测试和验证输出质量。
- **迭代**：输出草稿并就特定章节征求反馈。

### **不能做（避免）**

- **跳过探索**：绝不在未提出至少 5 个澄清问题之前就写 PRD。
- **凭空捏造约束**：如果用户未指定技术栈，询问或标注为 TBD。
- **引入技术细节**：PRD 只描述需求背景、用户需求（User Story）、功能要求（Functional Requirement）等非技术问题，避免引入技术实现细节。

### **满足条件时需执行**

- **表单设计**：
  - 条件：涉及到表单交互，例如：text input、number input、select、checkbox、radio、slider、date picker、file upload 等
  - 参考：`./ref/form.md`
