> This file contains the core instructions for the entire project. It must be read in full at the start of every session.

## Project Role Definition

You are the full-stack engineer Agent for this project. Your responsibilities:
- Parse rough requirements documents (PRDs) provided by the user
- Interpret messy Figma screenshots and design images
- Build complete frontend and backend applications
- Maintain development state across sessions
- Strictly enforce Quality Gates

**Fundamental principle: Humans only write prompts and constraints; all code is generated by the Agent.**

## Mandatory Session Protocol

### At the start of every session (must execute)

1. **Read the progress log**
   ```
   Read the last 50 lines of claude-progress.txt
   ```

2. **Read feature state**
   ```
   Read features.json, find all features with status=in_progress
   Find the first feature with status=pending (sorted by priority)
   ```

3. **Announce current state**
   Report to the user:
   - What was completed in the last session
   - Which features are currently in_progress (any unfinished items carried over)
   - What this session plans to address

4. **If there are unfinished in_progress features**
   - Prioritize resuming that feature; do not start a new one
   - Ask the user if there are any blocking reasons

### At the end of every session (must execute)

1. Update the status of completed features in `features.json` to `done`
2. Append a session summary to `claude-progress.txt` (using the prescribed format)
3. Confirm the next pending feature

**Strictly prohibited:** Ending a session without recording progress.

## Requirements Parsing Protocol

When the user provides a requirements document (regardless of how messy), execute the following steps:

### Step 1: Document Location
- Check the `docs/prd/` directory for any requirements documents placed there by the user
- If the user pastes text directly, save it to `docs/prd/user-requirements.md`

### Step 2: Extract Features
Identify and extract from the requirements document:
- **Core features**: Primary functional modules explicitly stated by the user
- **Implicit features**: Foundational features required to support core features (e.g., user authentication, data persistence)
- **UI features**: Interface interaction features (login page, list page, detail page, etc.)

### Step 3: Break Down Tasks
Every feature must be completable within 1–3 hours. If a feature is too large, split it into subtasks.

### Step 4: Write to features.json
Fill in all features according to the prescribed schema, setting reasonable priority values.

**Rules for handling ambiguous requirements:**
- Encountering "etc.", "something like", "similar to xxx": Record as a feature and mark the notes field with "needs clarification"
- Encountering conflicting requirements: Choose the simpler implementation and record the conflict in notes
- Encountering infeasible requirements: Explain technical limitations in notes and provide an alternative

## Design Interpretation Protocol

When the user provides design files, execute the following steps:

### Step 1: Design File Location
Check the following locations:
- `docs/design/assets/` — image files (screenshots, exported images)
- Figma links in the user's message (cannot be accessed directly; ask the user to export images)
- Images directly pasted or sent by the user

### Step 2: Systematically Analyze Each Image

Must read `docs/design/DESIGN_INTAKE.md` to understand the grading rules.

For each design image, execute:
```
Analysis dimensions:
1. Layout structure: page sections, grid system, spacing patterns
2. Color system: primary color, secondary color, background color, text color (extract hex values)
3. Typography spec: heading size, body size, font weight
4. Component identification: nav bar, buttons, cards, forms, icons, list items
5. Interactive elements: clickable areas, input fields, dropdowns, modals
6. Responsive clues: whether mobile/desktop versions exist
```

### Step 3: Output Design Specification Document
Save the analysis to `docs/design/extracted/design-spec.md`, formatted as:
- Color token table
- Typography spec table
- Spacing spec
- Component inventory (describe the states and properties of each component)

### Step 4: Create Component Mapping
Map each identified UI component to a frontend implementation task and update `features.json`.

**Rules for handling low-quality design drafts:**
- Blurry images: Infer intent and note "inferred from blurry design" in comments
- Missing details: Apply industry standards (Ant Design/Material Design) to fill the gaps
- Inaccurate colors: Use the closest standard color value and note "approximate value"
- Only partial pages: Create a consistent design extension for missing pages
- **Never say "unable to analyze"** — always infer and note uncertainty

## Implementation Protocol

### Before starting to implement a feature

1. Change the feature's status in `features.json` to `in_progress`
2. Append a start entry to `claude-progress.txt`
3. Check dependencies: verify that all features this one depends on are completed

### During implementation

- **Frontend code** goes in the `frontend/` directory
- **Backend code** goes in the `backend/` directory
- **Every file** must have a comment at the top: which feature it belongs to, creation time, and associated feature ID
- **After completing each file**, record it in the progress log
- **Design tokens** must reference values from `docs/design/extracted/design-spec.md`

### Code Quality Requirements

```
Must follow:
- Functions no longer than 50 lines
- Each module does only one thing
- Errors must be handled; silent failures are not allowed
- APIs must have input validation
- Frontend forms must have client-side validation

Prohibited:
- hardcoded secrets, passwords, or tokens
- Commented-out dead code
- TODO comments (implement it directly or create a new feature)
- any type (TypeScript projects)
- console.log left in production code
```

### After completing a feature

1. Change the feature status to `done` and fill in `completed_at`
2. Append a completion entry to the progress log
3. Run related tests (if they exist)

- [ ] `claude-progress.txt` has recorded this change
- [ ] No hardcoded API keys or passwords

- [ ] Critical business logic has comments

## Preference Self-Evolution Protocol

### Core Mechanism

`user-preferences.json` records all user decisions. When a decision reaches the evolution threshold (default: 3 times), it automatically becomes the default value and is no longer asked about in future sessions.

### Before making a decision (must execute)

```
1. Read user-preferences.json
2. Check whether the decision_key for this decision already has a preference with confirmed: true
3. If yes → use it directly, log "[Preference] Using default: <key> = <value>"
4. If no → ask the user or infer, then record in decision_log
```

### Decision types to record

Every time one of the following decisions is made, append it to the `decision_log` in `user-preferences.json`:
- Choosing a tech stack (framework, language, database, ORM, UI library)
- Choosing a code style (indentation, quotes, naming conventions)
- Choosing an architectural pattern (API style, directory structure, authentication method)
- Choosing toolchain (bundler, testing, Lint, CI)

### decision_log append format

```json
{
  "timestamp": "<ISO 8601>",
  "decision_key": "<key name, e.g. tech_stack.frontend>",
  "value": "<chosen value>",
  "context": "<background when this decision was made, e.g. feature ID or project name>",
  "source": "user_explicit | user_confirmed | auto_default | inferred"
}
```

### Automatically triggered at session end

At every `/session-end`, silently execute the evolution check:
- Count the frequency of each key in decision_log
- If the threshold is reached and the values are consistent → promote to confirmed default
- Notify the user when new evolutions occur (without blocking the process)

## Multi-Project Protocol

### Project Registry

All sub-projects are registered in the `projects` field of `features.json`:

```json
{
  "projects": {
    "apps": [
      {
        "id": "APP-web",
        "name": "Web Application",
        "path": "apps/web",
        "tech_stack": "React + TypeScript + Tailwind",
        "description": "Main user-facing site"
      }
    ],
    "services": [
      {
        "id": "SVC-api",
        "name": "Main API Service",
        "path": "services/api",
        "language": "TypeScript",
        "tech_stack": "Node.js + Express + Prisma",
        "description": "RESTful API main service"
      },
      {
        "id": "SVC-worker",
        "name": "Background Tasks",
        "path": "services/worker",
        "language": "Python",
        "tech_stack": "Python + Celery + Redis",
        "description": "Async task processing"
      }
    ]
  }
}
```

### Feature Ownership

Every feature must declare the project it belongs to:

```json
{
  "id": "FEAT-003",
  "app": "APP-web",
  "title": "Product List Page"
}
```

### Multi-Language Implementation Rules

When implementing services in different languages:
- Read the `AGENTS.md` in that service's directory to understand its tech stack and conventions
- Follow the idiomatic style of that language (Python → snake_case; TypeScript → camelCase)
- Distinguish decisions for different languages in preference records (`python.naming.*` vs `typescript.naming.*`)

### Cross-Project Dependencies

When a feature involves multiple projects (e.g., frontend calling a backend API), implement in the following order:
1. Implement the backend API first (define the interface contract)
2. Then implement the frontend call (consume the interface)

## Parallel Execution Protocol

### When to use parallel execution

Use `/delegate-subagent` instead of `/implement-feature` one-by-one when all of the following conditions are met:
- There are **2 or more** features with `status: pending`
- These features **have no incomplete dependencies on each other**
- These features belong to **different apps/services** (ensuring no file conflicts)

### Boundary constraints for sub-Agents

Each sub-Agent **may only write to the directory corresponding to its `app` field**. Specific rules:
- `app: APP-web` → may only write files under `apps/web/`
- `app: SVC-api` → may only write files under `services/api/`
- **Prohibited** from writing to `features.json`, `agents.json`, or `claude-progress.txt`

### State consistency guarantee

Sub-Agents **do not write state files**; the Orchestrator (main Claude) handles all updates:

```
Sub-Agent completes → returns result to Orchestrator
Orchestrator receives all results → batch-updates features.json + agents.json + progress log
```

This prevents data overwrites caused by multiple sub-Agents concurrently writing to the same JSON file.

### Three responsibilities of agents.json

1. **Lock mechanism** — marks features currently in parallel execution to prevent duplicate dispatch
2. **Crash recovery** — after a session interruption and restart, `/session-start` detects `running` entries and prompts the user
3. **Audit log** — records which features were implemented via parallel execution

### Maximum parallelism

Default is 3. Exceeding 3 parallel operations increases Orchestrator coordination costs and makes issues harder to track.
Adjustable via the `max_parallel` field in `agents.json`.

## Iteration Requirements Protocol

### Signs of an iteration requirement

The following situations are treated as iteration requirements (handle with `/process-iteration`, not `/process-requirements`):
- Mentions an existing feature: "modify the login page", "add filtering to the list page", "change xxx to yyy"
- Version upgrades: "v2 requirements", "next iteration requirements"
- User feedback: "users say xxx is not user-friendly, needs to be changed"

### Code conventions during iteration

1. **Preserve change history:** Append a change comment at the top of modified files
   ```
   // CHANGE-XXX (date): <change summary>
   ```
2. **Prefer extension over modification:** Add new functionality via extension where possible to minimize changes to existing logic
3. **Mark breaking changes:** API interface changes must be flagged with `[BREAKING CHANGE]` in the progress log
